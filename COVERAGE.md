## Analyzer quirks surfaced while building these

These are observations from making the apps detect — candidate
follow-ups for the analyzer, not bugs in the apps.

- **Complete flows are 0 across the board.** Cross-tier stitching
  resolves client→server edges (hence the partial flows), but no app
  produces a *complete* UI→caller→endpoint→DB flow. This matches the
  known gap (adorable issue #531: class-method dispatch in the call
  graph) — frontend interactions call a service method in another
  file/class, and the walker stops there. A good Phase-2 target.

- **`hasGoModule` / `hasCargoCrate` use exact-string matching on the
  manifest dependency key.** For versioned Go modules the import path
  carries a `/vN` suffix while several plugins' `appliesTo()` check the
  bare path:
  - `goredis` checks `github.com/redis/go-redis` but the import is
    `.../v9` (app 12 adds both lines to `go.mod`).
  - `chi` checks `github.com/go-chi/chi` but the import is `.../v5`
    (app 14 adds both lines).
  Worth normalizing `/vN` suffixes in the manifest matchers.

- **`grpcio` (Python) detects server servicers, not client stubs**
  (app 11) — the gRPC surface is modeled as a `Servicer` subclass.

- **`chi` `r.Route("/api", ...)` prefix composition** is not applied in
  this analyzer version (app 14) — routes are registered with full
  literal paths so client URLs stitch.

- **`.vue` / `.svelte` single-file components are not walked** — only
  `.ts/.tsx/...`. Vue/Svelte apps put component logic in `.ts` files
  (composition API / module scripts) to be seen by the analyzer.
